範例：會員制（狀態模式）

2020-08-21
Example

Pattern: 狀態模式

Class Diagram: 會員制

情境：平台有三種身份，分別是訪客 (guest) 、 會員 (member) 及 付費會員(premium)

  1. 訪客藉由註冊 (register) ，可成為會員。
  2. 會員藉由訂閱 (subsribe) ，可成為付費會員。
  3. 付費會員藉由取消訂閱 (cancelSubscription) ，可變回會員。
  4. 會員及付費會員藉由刪除帳號 (deleteAccount) ，可變回訪客。

隨著功能越來越多，每次新增功能時，
我們都會有許多複雜的條件式（判斷當前用戶狀態）。

讓我們用狀態模式改善它。

需求一：切分出不同的用戶狀態

  • 首先定義用戶狀態（抽象）
  • 訪客狀態
  • 會員狀態
  • 付費會員狀態
  • 最後修改原本的情境類別

[單一職責原則]
我們將情境類別與狀態類別視作兩種不同的職責。
透過委派來實現不同狀態下的行為。

[開放封閉原則]
修改既有狀態類別的行為，不會影響到全部的既有狀態類別。
（新增狀態類別時，可能會影響到）

[介面隔離原則]
情境類別依賴於抽象的用戶狀態 (User State) 。
不同的狀態類別實作抽象的用戶狀態 (User State) 。
